02一次播放的完整数据流

从用户点 ▶ 到耳朵听到声音，后端要走的全部环节。预热命中走 fast path，未命中走 cold path（约 10–15s）。

0
启动期（lifespan startup） prewarm_queue.start(default_voice, "song_intro")
后台 daemon 线程跑满 5 首待播 episode（推荐器 + 多源 + 缓存 + 稿件 + TTS 全流水），永远维持队列长度 5。预热失败不阻塞用户请求。
1
用户点 ▶ → GET /api/v1/episode?direction=next&voice=...&mode=...
前端 fetch 拿 episode payload。next 方向先 try_pop 预热队列。
2a
FAST PATH：预热命中（理想路径） prewarm_queue.try_pop(voice, mode, is_disliked)
命中即秒响应（~10ms），同步全局 _cursor + _recent_history，后台异步 refill 补一首。命中条件：voice/mode 匹配 + 未过期（< 30 分钟）+ dislike 未超阈值。
2b
COLD PATH：未命中（重新跑全流水） recommender.recommend_next(候选, recent, feedback, weather)
LLM 看候选歌单 + 最近 5 首 + 反馈 + 天气，选下一首。失败回退 cursor 顺序选歌。
3
多源调度：QQ 优先 → 网易兜底 qqmusic.get_track_by_songmid() → netease.get_track()
优先用年度报告解析出的 songmid 走 QQ 直链（VIP 月费覆盖 95%+）；QQ 失败/无 songmid 时网易兜底。两个都失败返回 track_not_found。
4
歌曲下载缓存（避开 20 分钟直链过期） song_cache.fetch_and_cache(source, source_id, audio_url)
流式写到 data/cache/songs/{source}_{id}.mp3。命中已有 cache 直接返回。
5
稿件：手写优先 → LLM 自动生成 llm._read_or_generate_script(entry, lyric, mode, environment)
先查 data/notes/scripts/<歌名>.md 手写稿；没有则按 mode 调 LLM（song_intro / classic_prose / history_today / quotation / weather_mood），生成后落盘做缓存。
6
Fish Audio TTS 合成 fish_audio.synthesize(script, voice_id)
按 (text, voice_id) 算 sha256 hash → 查 data/cache/tts/<hash>.mp3。命中瞬时；未命中调 Fish Audio API（5–10s）后落盘。
7
返回 episode payload（JSON）
字段包括 tts_url · song_url · song_title · artist · cover_url · lyric（LRC）· script · mode · degraded · recommend_reason。
8
前端：Web Audio 双轨混音 ctx.decodeAudioData() + 旁白 100% / 歌曲 15% 同步 start
DJ 时间对齐：解析 LRC 第一句歌词时间戳，让旁白结束 ≈ 歌曲第一句歌词出现。旁白结束后 3 秒线性渐升歌曲到 100%。
9
歌曲播完 → 自动续播 songSource.onended → scheduleAutoNext(2)
2 秒短暂停（电台主播切歌「喘气感」）→ 触发 nextEpisode → 回到流程 1。如果队列预热中已就绪，下一首 0 等待。

03模块矩阵 M1–M6

六大模块（与 bug.md 完全对齐）。每个模块单点失败时，邻近模块兜底降级。

M1编排

backend/main.py · backend/services/prewarm.py · recommender.py

episode 路由 + cursor 推进 + 预热队列 + 反馈跳过阈值（dislike ≥ 2 自动跳）。fast/cold 双路径合并，对前端是单一接口。

M2歌源

backend/adapters/qqmusic.py · netease.py · base.py

QQ 优先（按年度报告 songmid 直取）→ 网易兜底（搜索匹配）。Track 统一接口暴露 audio_url + cover_url + lyric。本地 cache 避开 20 分钟直链过期。

M3文案 & TTS

services/llm.py · llm_providers.py · fish_audio.py · listening_facts.py · facts.py

5 种 mode（song_intro / classic_prose / history_today / quotation / weather_mood）+ DeepSeek/MiMo 双 LLM provider 切换。Fish Audio TTS 按 hash 缓存。listening_facts 池注入「你的偏好」。

M4播放器

frontend/index.html · Web Audio API

AudioContext + 双 SourceNode + GainNode 混音引擎。DJ 时间对齐（LRC 第一句歌词时间戳跳过前奏）+ 进度条 + 上下首 + 自动续播。

M5UI

frontend/index.html（CSS + DOM）

极简底栏播放器 + 流动歌词卡片 + 配置面板 + 萤火虫粒子。深紫光斑暗色终端风（漂浮光斑 + 鼠标萤火虫粒子 + 霞鹜文楷 + 等宽终端排版）。V3.1 后已删氛围视频/呼吸效果，纯音频体验。

M6配置 & 数据

data/config.json · user/taste.md · feedback.jsonl · 年度报告截图/

所有 API key + 个人偏好 + 听歌历史。data/ 整体 .gitignore，敏感字段隔离。年度报告截图驱动 5 年 209 首歌单合并 + listening_facts 池。

06已知阻塞 & 待修

影响当前体验的活跃问题清单。

B003设置中配置的位置不生效

P1 · M6 · 配置 location 后仍显示自动定位结果，必须手动 refetch 才正确。配置保存后未触发位置 state 更新。

B004切下一首失败（✅ 已修复）

P1 · M4 · 点「下一首」无响应。根因：startPlayback catch 的 else 分支漏了恢复 nextBtn/prevBtn，已补齐；定位用的诊断日志已清理。

B005预留歌曲播放入口未实现

P2 · M4 · 占位状态，需先与所有者对齐功能定义。

QQ-AUTHQQ 音乐 cookie 失效（invalidq）

非 bug · 5/14 18:11 后 qm_keyst 被腾讯标记失效，CGI 全部 104009。所有 QQ 路径走兜底网易，但网易 vipType=0 大面积 404 → 体验全瘫。需重新登录 QQ 音乐网页版拿新 cookie 更新 config.json。

NETEASE-VIP网易云 vipType=0

非 bug · 网易账号无绿钻，2024+ 版权歌大概率拿不到直链。建议 (a) QQ 复活后由 QQ 主导；(b) 也充网易绿钻；(c) source_overrides.json 手工锁定。
