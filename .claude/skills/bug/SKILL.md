# /bug — Bug 文件管理

## 触发场景

- `/bug` —— 主入口。先输出当前 bug 概况，再询问要执行的操作
- `/bug <自由描述>` —— 默认理解为新增 bug，按描述启动新增流程
- `/bug new <描述>` —— 显式新增
- `/bug start <编号>` —— 把 bug 状态标记为「修复中」
- `/bug fix <编号>` —— 标记修复完成（实时取修复时间，迁到「已修复」分区）
- `/bug shelve <编号> <原因>` —— 搁置（迁到「已搁置」分区）
- `/bug list [状态]` —— 列出对应状态下的 bug，状态可选 `待修` / `修复中` / `已修复` / `已搁置`，默认 `待修`
- `/bug init` —— 强制重新初始化 bug.md（覆盖前必须二次确认）

## 写入位置

bug 文件唯一路径：`E:\AI电台项目\开发文件\bug.md`

## 时间格式约定（最关键的规则）

- **格式**：ISO 8601 — `YYYY-MM-DDTHH:MM:SS`，例如 `2026-04-28T20:37:56`
- **必须实时获取系统时间**——禁止硬编码、估算、复用旧时间戳，禁止从对话上下文里"读取"时间
- **同一会话内连续录入多条 bug，每条都必须各自重新调用获取命令**
- **获取命令**：
  - bash（Git Bash / WSL）：`date +"%Y-%m-%dT%H:%M:%S"`
  - PowerShell：`Get-Date -Format "yyyy-MM-ddTHH:mm:ss"`
- 涉及时间字段的所有操作（发现时间、修复时间、搁置时间）都必须先执行获取命令再写入

## 严重程度

| 等级 | 含义 | 推断关键词 |
|------|------|-----------|
| P0 | 阻塞使用 / 页面打不开 / 核心数据错误 | 打不开 / 阻塞 / 崩 / 白屏 |
| P1 | 核心功能失效但不影响其他功能 | 无法 / 不能 / 失效 / 不响应 |
| P2 | 体验问题 / 视觉异常 / 边界场景 bug | 丑 / 错位 / 偶尔 / 不好看 |
| P3 | 优化项 / 吹毛求疵 | 建议 / 可以更 / 小问题 |

用户没明确指定时按关键词推断，**推断结果必须口头向所有者确认后再写入**。

## 状态枚举

- `待修` — 已记录但未开始修
- `修复中` — 正在处理
- `已修复` — 修完了，必须在条目内补「修复时间」+「修复说明」
- `已搁置` — 暂不修，必须在条目内补「搁置原因」

## 编号规则

- 格式：`B` + 三位数字（B001 / B002 / B003 …）
- **全局递增**，跨三个分区计算最大值 + 1
- **不复用**已删除或已搁置条目的编号

## 字段填写规则

| 字段 | 何时填 | 来源 / 注意 |
|------|--------|-------------|
| 发现时间 | 新增时 | **实时获取**，ISO 8601 |
| 修复时间 | 状态改为已修复时 | **实时获取**，ISO 8601 |
| 搁置时间 | 状态改为已搁置时 | **实时获取**，ISO 8601 |
| 模块 | 新增时 | 优先用模块编号（M1 编排 / M2 歌源 / M3 文案&TTS / M4 播放器 / M5 UI / M6 配置&数据 / 全站） |
| 现象 | 新增时 | 用户原始描述，不要过度归纳 |
| 复现步骤 | 新增时 | 至少 1 步；无法稳定复现写「偶发，xx 时出现」 |
| 环境 | 新增时 | 浏览器 + 设备类型 + 屏幕尺寸；用户没说则口头询问，或填「待补充」 |
| 严重程度 | 新增时 | 按关键词推断 + 口头确认 |
| 修复思路 | 新增时（可选） | Claude 给初步思路；不强制，可写「待排查」 |
| 修复说明 | 状态改为已修复时 | 改了哪个文件、关键代码改动、commit hash（已建仓后） |
| 搁置原因 | 状态改为已搁置时 | 一句话写明为什么不修 |
| 状态 | 每次状态变更 | 与所在分区一致，不能错位 |

## 状态流转规则

**核心约定：状态字段值必须与所在分区一致，禁止错位。** 状态变更时必须把整个条目从原分区**整体迁移**到目标分区，不能只改字段。

- 新增 → 状态 = `待修`，放入「待修复」分区顶部
- 标记修复中 → 状态 = `修复中`，**仍在「待修复」分区**（用 ⚠️ 标识区分），便于看到正在处理的条目
- 标记修复完成 → 状态 = `已修复`，从原分区移除，插入「已修复」分区顶部
- 标记搁置 → 状态 = `已搁置`，从原分区移除，插入「已搁置」分区顶部

## 排序规则

- 「待修复」：P0 → P1 → P2 → P3，同级按发现时间倒序（新在上）；状态为「修复中」的条目在同级内**置顶**
- 「已修复」：按修复时间倒序(最新在上)
- 「已搁置」：按搁置时间倒序

## 操作流程（Claude 执行步骤）

### A. `/bug`（无参数）— 主入口

1. 用 Read 读 `E:\AI电台项目\开发文件\bug.md`
   - 文件不存在 → 跳到流程 G「文件初始化」
2. 解析三个分区，统计各状态条目数
3. 输出概况（中文回复给所有者）：

   ```
   📊 当前 bug 概况
   - 待修复：N 条（P0×x / P1×y / P2×z / P3×w，其中修复中 m 条）
   - 已修复：M 条
   - 已搁置：K 条
   ```

4. 询问要执行的操作（新增 / 修复 / 搁置 / 列表）

### B. 新增 bug（`/bug new <描述>` 或 `/bug <自由描述>`）

1. **用 Bash 实时获取发现时间**：`date +"%Y-%m-%dT%H:%M:%S"`，把输出原样保留
2. 推断严重程度（按关键词），**口头向所有者确认**
3. 询问 / 填写其他字段：模块、复现步骤、环境、修复思路
4. 用 Read 读 bug.md，扫描三个分区找出最大编号，新编号 = 最大 + 1
5. 按字段拼装条目（见下方「条目格式」）
6. 用 Read 读 bug.md → 内存中按排序规则插入到「待修复」分区相应位置 → 用 Write 整体重写
7. 回执：「✅ 已新增 B0XX [P?] xxx，发现时间 YYYY-MM-DDTHH:MM:SS」

### C. 标记修复中（`/bug start <编号>`）

1. 用 Read 读 bug.md，定位 B编号 条目
2. 把状态字段从 `待修` 改为 `修复中`
3. 在标题前添加 ⚠️ 标识（便于快速识别）
4. 在「待修复」分区内重排（修复中条目同级置顶）
5. 用 Write 重写文件
6. 回执：「⚠️ B0XX 已标记为修复中」

### D. 标记修复完成（`/bug fix <编号>`）

1. 用 Read 读 bug.md，定位 B编号 条目（应在「待修复」分区）
2. **用 Bash 实时获取修复时间**：`date +"%Y-%m-%dT%H:%M:%S"`
3. 询问「修复说明」（commit / 改了哪个文件 / 关键改动）
4. 在条目「发现时间」字段下插入「修复时间」字段
5. 在条目末尾追加「修复说明」字段
6. 状态字段改为 `已修复`，去掉 ⚠️ 标识
7. 把整个条目从「待修复」分区移除 → 插入到「已修复」分区顶部
8. 用 Write 整体重写文件
9. 回执：「✅ B0XX 已标记为修复完成，修复时间 YYYY-MM-DDTHH:MM:SS」

### E. 搁置（`/bug shelve <编号> <原因>`）

1. 用 Read 读 bug.md，定位条目
2. **用 Bash 实时获取搁置时间**：`date +"%Y-%m-%dT%H:%M:%S"`
3. 在条目末尾追加「搁置时间」+「搁置原因」字段
4. 状态字段改为 `已搁置`
5. 把条目从原分区移到「已搁置」分区顶部
6. 用 Write 整体重写文件
7. 回执：「📦 B0XX 已搁置（原因：xxx）」

### F. 列出（`/bug list [状态]`）

1. 用 Read 读 bug.md
2. 默认列出「待修复」分区；若指定状态参数则列对应分区
3. 输出列表（每行一条）：

   ```
   B001 [P1] ⚠️修复中 简短标题（模块：M5）— 发现 2026-04-28T20:37:56
   B002 [P2] 待修 简短标题（模块：M3）— 发现 2026-04-28T19:12:33
   ```

### G. 文件初始化（`/bug init` 或文件不存在）

1. 检查 bug.md 是否存在
   - **存在**：必须二次确认「确认覆盖现有 bug.md？所有内容会丢失」，得到明确同意才继续
   - **不存在**：直接进入下一步
2. 用 Write 创建 bug.md，内容见下方「文件初始化模板」
3. 回执：「✅ 已初始化 bug.md，可以开始录入 bug」

## 注意事项

- **时间禁止硬编码**：每次涉及发现时间 / 修复时间 / 搁置时间，必须实时执行获取命令；禁止从对话上下文里"读取"时间，禁止"刚刚那个时间复用一下"
- **状态与分区必须一致**：已修复的 bug 必须**整体迁出**「待修复」分区，禁止只改状态字段不迁移
- **编号不复用**：跨三个分区计算最大编号 + 1
- **严重程度不静默推断**：按关键词推断后必须口头向所有者确认
- **写入失败立即提示**：权限错误、路径错误、文件被外部锁定等，不要静默重试
- **覆盖式写入前要求确认**：`/bug init` 命令在文件已存在时必须二次确认
- **保护现有内容**：每次状态流转前用 Read 读取最新文件再修改，避免基于过时内容写回造成丢失
